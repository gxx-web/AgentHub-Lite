# AgentHub-Lite 数据模型设计方案

## 1. 设计目标

本方案根据根目录 `README.md`、`docs/api.md` 以及当前 FastAPI / React 代码中的数据结构设计，目标是先支撑 MVP 的用户登录、工作区、模型供应商、Agent 配置、提示词和会话调试闭环，同时为知识库、Skill、MCP、任务调度、审计日志等后续能力预留扩展空间。

配套 SQL 文件：

```text
infra/postgres_schema.sql
```

执行方式：

```bash
psql "postgresql://agenthub:agenthub@localhost:5432/agenthub" -f infra/postgres_schema.sql
```

## 2. 设计原则

1. 工作区隔离：核心业务表都通过 `workspace_id` 归属到工作区，后续支持个人和团队数据隔离。
2. 字符串 ID 兼容：当前 API 使用 `default-agent`、`openai-compatible` 这类字符串 ID，数据库继续使用 `text` 主键，降低迁移成本。
3. 配置结构化：稳定字段单独建列，例如 `model_name`、`temperature`；变化快的能力配置使用 `jsonb`，例如 `memory_config`、`retrieval_config`。
4. 敏感信息拆分：模型 API Key 放在 `model_credentials`，和 `model_providers` 分表，便于后续加密、密钥托管和权限控制。
5. 多对多关系显式建表：Agent 与知识库、Skill、MCP Tool 使用关联表，避免长期依赖数组字段造成查询和权限控制困难。
6. 运行过程可追踪：会话、消息、Agent Run、Tool Call、审计日志单独建表，支撑调试、统计和问题回放。

## 3. 核心实体

### 3.1 用户与工作区

`users` 存储用户账号信息，包括邮箱、用户名、密码哈希、显示名、状态和最后登录时间。

`workspaces` 存储个人或团队工作区。MVP 可以只使用默认工作区 `default`。

`workspace_members` 存储用户与工作区的成员关系，角色包括：

- `owner`
- `admin`
- `member`
- `viewer`

当前前端账号密码仍是硬编码，后续接后端鉴权时，登录接口应使用 `users.username` 或 `users.email` 查询用户，再校验 `password_hash`。

### 3.2 模型供应商

`model_providers` 对应当前后端 `ModelProviderConfig`：

- `id`
- `name`
- `provider_type`
- `base_url`
- `default_model`
- `available_models`
- `supports_streaming`
- `supports_tool_calling`

`model_credentials` 保存模型密钥元数据：

- `secret_ref`：外部密钥系统引用。
- `encrypted_api_key`：本地加密后的 API Key。

生产环境建议优先使用 `secret_ref` 或应用层加密，不要明文保存 API Key。

### 3.3 提示词

`prompts` 存储可复用提示词模板，包括系统提示词、用户提示词、工具提示词和工作流提示词。

`prompt_versions` 存储提示词版本。Agent 可以绑定 `prompt_id`，同时在 `agents.system_prompt` 中保留运行时快照，避免提示词被修改后历史 Agent 配置不可复现。

### 3.4 Agent

`agents` 对应当前前端和后端的 Agent 字段：

- 基础信息：`name`、`description`、`avatar_url`
- 提示词：`prompt_id`、`system_prompt`
- 模型配置：`model_provider_id`、`model_name`、`temperature`、`top_p`、`max_tokens`
- 记忆配置：`memory_config`
- UI 引导：`opening_message`、`example_questions`
- 发布状态：`visibility`、`status`

`agent_versions` 存储 Agent 版本快照，适合后续支持克隆、回滚、导入导出和发布历史。

Agent 扩展绑定：

- `agent_knowledge_bases`
- `agent_skills`
- `agent_mcp_tools`

这些关联表用于支持一个 Agent 绑定多个知识库、多个 Skill 和多个 MCP 工具。

### 3.5 会话与消息

`conversations` 对应当前后端 `ConversationRead`，用于记录某个用户和某个 Agent 的一次对话。

`messages` 对应当前 `ChatMessage`：

- `role` 支持 `system`、`user`、`assistant`、`tool`
- `content` 保存消息正文
- `metadata` 保存工具调用、引用片段、前端展示状态等扩展信息

当前前端按照 `conversation.messages` 展示消息。后续改为数据库查询时，可以按 `conversation_id, created_at` 升序读取。

### 3.6 运行日志

`agent_runs` 记录每次 Agent 调用：

- 输入输出
- 使用的模型供应商和模型
- Token 用量
- 延迟
- 成本
- 错误信息
- trace

`tool_calls` 记录某次 Agent Run 中的工具调用过程，包括工具名、输入、输出、状态和耗时。

这两张表是后续调试、账单统计、错误排查和审计的基础。

## 4. 扩展实体

### 4.1 知识库

`knowledge_bases` 记录知识库配置，`backend` 默认是 `ragflow`，`external_id` 用于关联 RAGFlow 侧知识库 ID。

`knowledge_documents` 记录上传文档、解析状态和外部文档 ID。

### 4.2 Skill

`skills` 存储可复用能力包，`manifest` 使用 `jsonb` 保存 Skill 的结构化描述、参数、资源清单和执行配置。

### 4.3 MCP

`mcp_servers` 存储 MCP Server 连接配置，支持 `stdio`、`http`、`sse`。

`mcp_tools` 存储从 MCP Server 发现的工具清单和输入 schema。

### 4.4 定时任务与通知

`scheduled_tasks` 用于后续把 Agent 配置为 Cron 或固定间隔任务。

`notifications` 保存邮件、Webhook、企业微信、飞书、钉钉、Telegram 等通知渠道配置。

### 4.5 API Key 与审计

`api_keys` 用于 Agent API 发布后的调用鉴权，只保存 `key_hash` 和 `key_prefix`，不保存明文 key。

`audit_logs` 记录用户操作，包括操作类型、资源类型、资源 ID、IP、User-Agent 和详细内容。

## 5. 主要关系

```text
users 1 --- n workspace_members n --- 1 workspaces
workspaces 1 --- n model_providers
model_providers 1 --- 1 model_credentials
workspaces 1 --- n prompts
prompts 1 --- n prompt_versions
workspaces 1 --- n agents
prompts 1 --- n agents
model_providers 1 --- n agents
agents 1 --- n conversations
conversations 1 --- n messages
agents 1 --- n agent_runs
agent_runs 1 --- n tool_calls
agents n --- n knowledge_bases
agents n --- n skills
agents n --- n mcp_tools
mcp_servers 1 --- n mcp_tools
```

## 6. MVP 落地建议

第一阶段优先接入这些表：

1. `users`
2. `workspaces`
3. `workspace_members`
4. `model_providers`
5. `model_credentials`
6. `prompts`
7. `agents`
8. `conversations`
9. `messages`
10. `agent_runs`

对应当前代码改造顺序：

1. 新增数据库连接配置和 SQLAlchemy / SQLModel 基础设施。
2. 将 `AgentService` 的内存字典迁移到 `agents`、`prompts`。
3. 将 `ModelProviderService` 的内存字典迁移到 `model_providers`、`model_credentials`。
4. 将 `ChatService` 的内存会话迁移到 `conversations`、`messages`、`agent_runs`。
5. 新增用户登录接口，把当前前端硬编码登录替换为后端鉴权。

## 7. 默认种子数据

SQL 文件内置以下开发数据：

- 用户：`user-admin`
- 用户名：`admin`
- 工作区：`default`
- 模型供应商：`openai-compatible`
- 默认提示词：`prompt-default-agent-system`
- 默认 Agent：`default-agent`
- 默认会话：`default-conversation`
- 欢迎消息：`welcome-message`

注意：SQL 中的 `password_hash` 是开发占位值，不是可用于生产的密码哈希。真正接入登录时，需要由后端使用 bcrypt、argon2 等算法生成并校验密码哈希。

## 8. 安全与生产注意事项

1. 密码只保存哈希，禁止保存明文。
2. 模型 API Key 必须加密或托管到密钥管理系统。
3. API Key 表只保存 hash 和 prefix，不保存明文 token。
4. 审计日志要避免记录完整密钥、密码、Authorization Header 等敏感数据。
5. 多工作区接口查询必须带 `workspace_id` 权限过滤。
6. 删除用户时，业务数据建议保留并将 `user_id` 置空；删除工作区时再级联删除工作区内资源。

## 9. 后续迁移建议

当前 SQL 是初始化脚本。进入持续开发后，建议引入 Alembic：

```text
infra/migrations/
```

每次表结构变更生成一条迁移，避免直接修改生产数据库初始化脚本。
